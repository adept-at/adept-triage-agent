Skills Store & Triage Pipeline

End-to-end flow showing how the skills store provides institutional memory to classification and repair, and how every fix attempt feeds back into the store as a learning loop.

The skills store is the agent’s long-term memory. It persists fix patterns to DynamoDB, then surfaces them as context in the classifier prompt, investigation prompt, and every repair agent’s prompt — with per-agent framing to avoid anchoring bias. Skills auto-retire when they stop working.

3. Skill Store Initialization

src/pipeline/coordinator.ts — Constructs store, loads existing skills
Construction
When AUTO_FIX_TARGET_REPO resolves (defaults to the current repo), the coordinator constructs the store:

new SkillStore(region, tableName, owner, repo)

AWS credentials are supplied via the standard provider chain (OIDC-configured role, static keys, etc.). If the load() query fails, the run continues with an empty in-memory cache and loadSucceeded stays false so save() knows to skip pruning.

skillStore.load()
QueryCommand {
  TableName: "triage-skills-v1-live",
  KeyConditionExpression: "pk = :pk AND begins_with(sk, :prefix)",
  ExpressionAttributeValues: {
    ":pk": "REPO#adept-at/wdio-9-bidi-mux3",

→ Loaded 12 skill(s) from DynamoDB
→ Each skill backfilled: successCount=0, failCount=0, retired=false, ...
Non-fatal on failure. If DynamoDB is unreachable, load() warns, marks loaded=true to avoid retry thrash, and keeps loadSucceeded=false. The run continues with an empty in-memory cache and any save() during the run writes to Dynamo best-effort while skipping the prune step (we cannot safely pick oldest skills against an incomplete view of the table).

4. Skill-Informed Classification

src/pipeline/coordinator.ts classify() — Skills injected into classifier prompt before the LLM call
Step A: Flakiness Detection
skillStore.detectFlakiness("lms.video.plays.e2e.ts")

Scans all skills for this spec. Triggers if:
• >1 fix in 3 days (short window) — "chronically flaky"
• >2 fixes in 7 days (long window) — "recurring instability"

Result: { isFlaky: true, fixCount: 3, windowDays: 7, message: "auto-fixed 3 times in 7 days" }
Step B: Find Classifier Skills
skillStore.formatForClassifier({ framework, spec, errorMessage })

findForClassifier() filters:
• Same framework OR unknown — framework match
• retired === false — retired excluded
• validatedLocally === true — proven fixes only

Scoring: spec match (+15) + error similarity × 5 + recency <7d (+3)
Returns top 3, sanitized for prompt injection.
What gets injected into the OpenAI classifier prompt

5-Agent Pipeline (with skill context throughout)
Analysis Agent    → classifies root cause (sees skill context)
Code Reading      → fetches test file + page objects (no LLM)
Investigation     → cross-references with code (sees investigation context)
Fix Generation    → generates oldCode/newCode (sees fix patterns)
Review Agent      → approves or rejects (sees success track records)
  ↓
If rejected → Fix Gen again with review feedback (up to 3 iterations)
↓
fix approved → local validation loop: clone, apply patch, run test

6. Local Validation & PR

src/services/local-fix-validator.ts & src/pipeline/validator.ts — Up to 3 iterations
Iteration 1:
  1. Clone repo, npm ci (with @actions/cache for speed)
  2. Baseline check — run test WITHOUT fix
     • If baseline passes → abort (failure was transient, no fix needed)
     • If baseline fails → continue
  3. Apply patch: find oldCode in file, replace with newCode
  4. Run: VALIDATION_TEST_COMMAND with {spec} and {url} substituted
  5. If PASS → git push + pulls.create() → PR opened
  6. If FAIL → reset files, feed test logs back to repair pipeline

Iteration 2: same 5-agent pipeline with previous failure context
Iteration 3: last attempt (duplicate fix fingerprint stops early)
↓
fix attempt complete (pass or fail) → now persist the learning back to the skill store

8. Skill Lifecycle Management

Automatic retirement and pruning — the store is self-healing
Auto-Retirement
Triggered inside recordOutcome() after every success/failure update.

failRate = failCount / (successCount + failCount)

Retire when failRate > 40% AND failCount ≥ 3

Retired skills are excluded from all queries — findRelevant(), findForClassifier(), and formatSkillsForPrompt() all filter retired === false. 
Example: Skill e5f6g7h8
  successCount: 2, failCount: 4 → failRate = 67%
  → 67% > 40% AND 4 ≥ 3
  → UPDATE SET retired = true
  → "⚠️ Skill e5f6g7h8 retired — 67% failure rate"
Auto-Pruning (Cap at 100)
Triggered inside save() when total skills exceed MAX_SKILLS (100).

DynamoDB Sorts by createdAt (oldest first), protects the just-saved skill, deletes overflow via DeleteCommand with condition expressions.

Git Slices the array to keep the newest 100: skills.slice(-MAX_SKILLS)
↓
next failure of this spec → the new skill is now available for classification and repair

Appendix: Query Methods

src/services/skill-store.ts — 5 read methods, each with specific filtering and scoring
findForClassifier()
Used by: Classification (step 4)
Filters: non-retired, validated locally, framework match
Scoring: spec (+15), error similarity (×5), recency <7d (+3)
Returns: top 3
findRelevant()
Used by: Investigation context (step 5)
Filters: non-retired, framework match (includes unvalidated)
Scoring: spec (+10), error similarity (×5)
Returns: top 5
detectFlakiness()
Used by: Classification (step 4) + outputs
Logic: count skills for spec in time windows
Thresholds: >1 in 3 days OR >2 in 7 days
Returns: FlakinessSignal { isFlaky, fixCount, windowDays, message }
formatForClassifier()
Used by: OpenAI classifier prompt (step 4)
Calls: findForClassifier() then formats each skill as numbered list
Fields: errorPattern, rootCauseCategory, fix.summary, confidence
Safety: all text through sanitizeForPrompt()
formatForInvestigation()
Used by: Repair pipeline investigation context (step 5)
Calls: findRelevant() filtered to skills with investigationFindings
Fields: findings, rootCauseChain, classificationOutcome, repoContext
Returns: top 3 with date and outcome
Backend: DynamoDB (triage-skills-v1-live) · Cap: 100 skills per repo · Auto-retire: >40% fail rate with ≥3 failures · Model: gpt-5.3-codex · Runtime: Node.js 24
