# v1.51.1 — Extraction Quality Hardening

> **Status:** Design — pre-implementation. No code changes until this doc is signed off.
>
> Targeted follow-up to v1.51.0. Focuses on two observed evidence-extraction gaps that caused the agent to anchor on background console noise rather than the test's actual failure signal.

---

## Philosophy

v1.51.0 upgraded the two agents where reasoning observably fell short (fix-gen + review → gpt-5.4 + xhigh). v1.51.1 addresses an orthogonal quality issue: even with a stronger reasoning model, the agent can only be as good as the evidence it receives. On a recent lib-cypress-canary run against PR #3650, the agent correctly classified the failure as PRODUCT_ISSUE with 95% confidence, but its summary latched onto `Error: Cannot query field "getChannelsForUser"` — a cross-cutting browser console error from the v1.50.x communication-refactor merge that now fires on every page load, not the test's actual failure mode (a 403 from the content API when attaching objects to a skill).

Two targeted tightens to the evidence pipeline:

1. **Classifier prompt rule: distinguish CAUSAL errors from BACKGROUND console noise.** One instruction in the classifier system prompt. ~15 LOC. High leverage because the LLM already has the right context; it just needs steering on how to rank it.

2. **Reject HTTP(S) URLs in file-attribution patterns.** Long-standing follow-up from the learn-webapp PR #3608 audit. Stack-trace patterns like `/at\s+.+?\((.+?\.(js|ts|jsx|tsx)):\d+:\d+\)/` accept URLs such as `https://learn-webapp-*.vercel.app/static/js/vendor.axios.*.js:line:col` as the "test file," which poisons the File: attribution in the Slack output and breaks CodeReadingAgent's fetch. Reject URLs explicitly.

### Safety invariants (non-negotiable)

1. **No prompt rewrites.** One new paragraph added to the existing classifier system prompt, sandwiched between the CROSS-BROWSER HARD RULE (v1.48.2) and the COMMON MISCLASSIFICATION PATTERNS section. Does not touch existing rules.
2. **Single-commit rollback.** Both changes are localized and trivially revertible.
3. **No new action inputs.** No new constants. No new env vars.
4. **Test suite unchanged.** 643/643 must still pass. No new tests.
5. **Bit-exact for non-matching cases.** URL rejection only alters behavior when `fileName` WOULD have been a URL. For logs with a valid relative spec path extracted cleanly, behavior is unchanged.

## Change 1 — Classifier prompt rule

### Where

The classifier system prompt lives in `src/openai-client.ts`. A new section is added between the existing CROSS-BROWSER EVIDENCE HARD RULE section and the COMMON MISCLASSIFICATION PATTERNS section (roughly after line 238 and before line 240 in the current file).

### What to add

```
CAUSAL SIGNAL vs BACKGROUND NOISE (CRITICAL):

Logs from a failed test often contain two distinct classes of error:
1. The CAUSAL error — what the test framework itself asserted against, timed out on, or otherwise declared the failure around. This has a direct causal link to the test's failure.
2. BACKGROUND errors — console errors, XHR failures, or GraphQL errors that fired during the run but were not what the test framework was waiting on. These are often cross-cutting noise from unrelated product code that runs on every page load.

When these diverge, anchor your verdict on the CAUSAL error:
- Cypress frames its failure in a "N failing" block followed by the test name, the assertion message, and a stack trace. WDIO frames it as "FAILED in ..." followed by a stack trace. Whatever the test framework declares as the failure IS the causal signal.
- Background errors — however loud, however hard-typed, however many times they repeat — are context, not cause. They may indicate real product bugs in other areas, but they are not necessarily what killed THIS test.
- If the only hard-typed error you can identify is a browser console error with no clear link to what the test framework waited on, state that uncertainty explicitly. Prefer INCONCLUSIVE over committing to a PRODUCT_ISSUE verdict based on a console error that may be unrelated to the test's actual failure point.

RED FLAG: if your reasoning anchors on an error that happens on EVERY page load (e.g., a console error from a component that always mounts, a GraphQL query that fires in a shared layout), that error is almost certainly NOT the cause of one specific test failing. It's background noise that predates the failure. Frequency in the logs that is disproportionate to a single test's scope (e.g., the error fires many times during navigation, page load, page unload, or before the test begins exercising the feature under test) is a strong indicator of cross-cutting background noise, not test-specific causal evidence.

RULE PRECEDENCE: if this rule appears to conflict with the CAUSAL CONSISTENCY RULE below (your hypothesis must be consistent with the PR diff), the test framework's own declared failure takes precedence for the verdict. When the PR diff correlates with one error and the framework's failure declaration points at a different error, state both in your indicators and reasoning, but anchor the verdict on the framework's failure point. The framework's failure is a scoped, test-specific signal; PR-diff correlation is circumstantial.

FALLBACK for crashed frameworks: if the test framework exited without declaring a specific failure (e.g., OOM kill, Cypress runner force-killed, process exit before the "N failing" block was written), there is no causal anchor to apply this rule to. In that case fall back to the infrastructure/INCONCLUSIVE path rather than promoting background errors to causal status.
```

### Why this wording

- **"CAUSAL vs BACKGROUND"** gives the LLM a clear two-category mental model rather than a vague "weigh signals" instruction.
- **Framework-specific cues** (Cypress "N failing" block, WDIO "FAILED in ...") are named explicitly so the LLM knows what shape to look for.
- **"Red flag"** pattern matches the existing prompt's voice (see lines 240-246 "COMMON MISCLASSIFICATION PATTERNS TO AVOID").
- **Frequency-disproportionate-to-scope signal** replaces an earlier cross-spec-recurrence phrasing that was unreliable. A single classifier call only sees ONE spec's ErrorData, so "same error across specs" is not reliably observable from within a single call — but "this error fires on every page load / before the test began / N times during navigation" IS observable from a single spec's log window and is the same underlying tell.
- **Rule precedence clause** resolves the known edge case where CAUSAL CONSISTENCY RULE (PR diff must support hypothesis) and CAUSAL vs BACKGROUND NOISE (anchor on framework failure) pull in different directions.
- **Framework-crashed fallback** handles the case where the framework exited before declaring a specific failure (OOM, force-kill, process crash) — no causal anchor is available, so falling back to INCONCLUSIVE/infrastructure is the correct behavior rather than promoting a background error to causal status by default.
- **Escape valve to INCONCLUSIVE** — explicit permission to say "I don't know" rather than being forced into a confident-but-wrong verdict. Matches the design pattern we already apply for cross-browser cases.

### What it does NOT say

- **Does not weaken the cross-browser HARD RULE.** That rule stays authoritative above this one.
- **Does not tell the LLM to ignore console errors.** Console errors can be causal (e.g., an uncaught exception that broke the page right before the test ran). The rule is about RANKING, not discarding.
- **Does not name specific error patterns to ignore.** `getChannelsForUser` is a today-specific example; six months from now the cross-cutting-noise pattern will be different. The rule describes the semantic shape, not the literal strings.

## Change 2 — URL rejection in file-attribution patterns

### Where

Two files:

1. `src/simplified-analyzer.ts` — the `filePatterns` loop inside `extractErrorFromLogs` (around line 286-302).
2. `src/agents/code-reading-agent.ts` — `cleanFilePath` (around line 273-298) and `extractFilePathsFromError` (around line 303-327).

### What to change

**In `simplified-analyzer.ts`:**

Add a URL rejection that considers ALL matches of each pattern before advancing to the next pattern. The existing code uses `.match()` which returns only the first match; if that first match is a URL-like path and we reject it, valid spec paths later in the same string are silently lost (e.g., `at getRet (https://...vercel.app/bundle.js:1:2)\n at Context.eval (cypress/e2e/spec.ts:10:20)` would drop the valid `cypress/e2e/spec.ts` path entirely). The fix uses `matchAll()` to iterate all matches per pattern:

```typescript
const isUrlOrBundlePath = (path: string): boolean => {
  if (!path) return true;
  const lower = path.toLowerCase();
  return (
    lower.startsWith('http://') ||
    lower.startsWith('https://') ||
    lower.startsWith('//') ||
    lower.includes('/__cypress/runner/') ||
    lower.includes('/static/js/vendor.')
  );
};

/**
 * Try all matches of each filePattern against the source, taking the
 * first non-URL/non-bundle match. Using matchAll (vs .match) is
 * required: .match returns only the first match, so if a stack trace
 * emits a URL frame BEFORE a valid spec frame, .match would hand back
 * the URL, we would reject it, and the valid spec path would be lost.
 */
const findValidSpecPath = (source: string): string | undefined => {
  for (const filePattern of filePatterns) {
    const globalPattern = filePattern.flags.includes('g')
      ? filePattern
      : new RegExp(filePattern.source, filePattern.flags + 'g');
    for (const m of source.matchAll(globalPattern)) {
      const candidate = m[1] || m[0];
      if (candidate && !isUrlOrBundlePath(candidate)) {
        return candidate;
      }
    }
  }
  return undefined;
};

const fileName = findValidSpecPath(errorContext) ?? findValidSpecPath(cleanLogs);
```

The `/static/js/vendor.` fragment catches bundled library paths (axios, react, etc.) served from CDN-like URLs. Not all bundle paths; the common ones that have poisoned past runs. Matches `/__cypress/runner/` in either URL or absolute-path form — a real test spec unlikely to ever live at this path in our onboarded repos.

**In `code-reading-agent.ts`:**

At the top of `cleanFilePath`, before any other logic. **Keep the URL-rejection predicate in sync with `simplified-analyzer.ts`'s `isUrlOrBundlePath`** (including the `//` protocol-relative prefix) so the two files agree on what counts as an unreachable path:

```typescript
private cleanFilePath(rawPath: string): string {
  if (!rawPath) return rawPath;

  // Reject HTTP(S) URLs, protocol-relative URLs, and Cypress runner
  // bundles. These commonly appear in stack traces from the browser
  // (Vercel previews, webpack output) and cannot be fetched from the
  // GitHub API. Returning empty lets the fallback chain in execute()
  // try the error-referenced-files path.
  //
  // Predicate must stay in sync with isUrlOrBundlePath in
  // simplified-analyzer.ts so both extractors reject the same class
  // of inputs.
  const lower = rawPath.toLowerCase();
  if (
    lower.startsWith('http://') ||
    lower.startsWith('https://') ||
    lower.startsWith('//') ||
    lower.includes('/__cypress/runner/') ||
    lower.includes('/static/js/vendor.')
  ) {
    return '';
  }

  // ... existing patterns ...
}
```

And in `extractFilePathsFromError`, add the same rejection inside the match loop:

```typescript
for (const pattern of patterns) {
  let match;
  while ((match = pattern.exec(errorMessage)) !== null) {
    const raw = match[1] || match[0];
    const cleaned = this.cleanFilePath(raw);
    // cleanFilePath now returns '' for URL/bundle paths; skip those.
    if (cleaned && !seen.has(cleaned)) {
      seen.add(cleaned);
      paths.push(cleaned);
    }
  }
}
```

Since `cleanFilePath` now returns empty for URLs, the existing `if (cleaned && ...)` guard handles it correctly. The only change is cleanFilePath itself; `extractFilePathsFromError` continues to work unmodified. The code-sketch above is illustrative of what will happen; the actual diff is just cleanFilePath.

### Why reject rather than clean

For `https://foo.vercel.app/static/js/vendor.axios.123.js`, in principle we could strip the host and try `static/js/vendor.axios.123.js` as a path. But:
- The file doesn't exist in the GitHub repo (it's a build output)
- The filename is content-hashed, so it's per-deploy-unique
- Attempting to fetch it always fails, burning API quota

Rejecting outright and letting the fallback-to-error-referenced-files path fire is cleaner.

## Risks and mitigations

### R1 — Overly-aggressive URL rejection
A legitimate test spec path that happens to contain `http://` inside a filename (e.g., `cypress/e2e/http-handler.spec.ts`) would not trip the rejection because the check is `startsWith('http://')`, not `includes`. The more aggressive `/__cypress/runner/` and `/static/js/vendor.` checks could theoretically false-positive if a repo has a test path like `src/__cypress/runner/mock.ts`. None of our onboarded repos have such paths; if one surfaces, tighten the pattern.

### R2 — Prompt-rule regression on legitimate console-error product bugs
A real product bug DOES sometimes show up as a console error (uncaught TypeError on page load, module resolution failure). The new rule ranks background errors below the framework's declared failure when both are present, and explicitly recommends INCONCLUSIVE when a console error is the ONLY evidence and cannot be linked to what the test framework waited on. That second clause is stronger than pure ranking — it actively steers the classifier away from PRODUCT_ISSUE on console-only evidence.

**Mitigation:** the rule's INCONCLUSIVE recommendation is scoped to "the only hard-typed error you can identify is a browser console error with no clear link to what the test framework waited on." When the framework DID emit a clear failure AND there's a console error, both are considered — the rule just instructs which to anchor on. When NO framework failure is clear and only a console error exists, INCONCLUSIVE is the honest verdict; a real product-code crash on page load typically ALSO produces framework-level symptoms (Cypress "failed to load the app" or similar) which would anchor the verdict without relying on the console error alone.

Canary measures whether PRODUCT_ISSUE rate drops unexpectedly on repos where console errors are legitimately causal. If metric #5 (validated-fix rate) regresses by ≥ 5pp, revert just the prompt change in v1.51.2 and keep the URL rejection.

### R3 — Prompt bloat
The classifier system prompt is already long (~300+ lines total once you account for the few-shot examples, HARD RULEs, and misclassification patterns). This adds ~30 lines / ~300-350 tokens per classifier call.

At 50-200 runs/day, additional cost per month: negligible (<$1).

### R4 — Interaction with v1.51.0 upgrades
Classification stays on `gpt-5.3-codex` in v1.51.0. This rule change runs against the same model. If v1.52.0 moves classification to `gpt-5.4`, the rule may need re-tuning for the newer model's instruction-following behavior. Out of scope for v1.51.1.

## Alternatives considered (not chosen for this release)

### Cypress "N failing" structural parser (audit improvement #2)

Would give framework-level failures a dedicated extraction path that outranks all generic `Error:` matches. Higher confidence than prompt-side guidance. Deferred because the prompt change is lower-risk and likely captures 80% of the value; only escalate to structural parsing if the prompt doesn't move CE1.

### 4xx HTTP signal elevation (audit improvement #5)

Would have surfaced the 403 on content-api in the PR #3650 case as a first-class signal. Deferred because 4xx is genuinely ambiguous (auth bugs, rate limits, validation errors can all be test-side). Needs careful domain-aware pattern design; not worth the risk for v1.51.1.

### Cross-spec aggregation (audit improvement #7)

Would detect "4 of 4 specs failed with the same error" as a deploy-level signal. Biggest ceiling, largest surface. Deferred to a separate release if the prompt rule's cross-spec awareness (via the RED FLAG paragraph) isn't enough.

### Low-evidence flag when artifact logs are tiny (audit improvement #4)

Would annotate runs with artifact logs < 1KB as low-evidence. Would have helped the PR #3650 case where only 406 bytes of artifact logs arrived. Candidate for v1.51.2 if prompt rule alone doesn't move CE1/CE2.
